zcli

zcash wallet CLI - ssh keys as wallet seed

deterministic zcash wallet derived from ed25519 ssh keys. built for agents - any process with ssh access already has a wallet. no key ceremony, no mnemonic management, no interactive setup. same key, same wallet, same addresses, every time.

why agents

agents authenticate with ssh keys. zcli derives a zcash spending key from the same ed25519 seed - one key for both identity and money. no new secrets to provision, no seed phrases to store, no interactive prompts. a headless process can sync, check balance, and send transactions with a single binary and the key it already has. 

ZCLI_IDENTITY=~/.ssh/id_agent zcli balance --json
{"transparent":0,"shielded":50000,"total":50000,"total_zec":"0.00050000"}

every agent gets a unique wallet from its ssh key. deterministic derivation means no state to back up beyond the key itself.

usage

zcli -i ~/.ssh/id_ed25519 address          # show addresses
zcli sync                                   # scan chain
zcli balance                                # check funds
zcli send 0.01 u1...                        # send shielded
zcli shield                                 # t-addr → shielded
zcli receive                                # QR code
zcli notes --json                           # list received notes
zcli export                                 # export viewing key
zcli board --port 3333                      # sync loop + JSON API

all commands accept --json for machine-readable output. set ZCLI_IDENTITY, ZCLI_ENDPOINT as env vars for headless operation.

architecture

zcli

the wallet. derives orchard spending keys and transparent keys from an ssh ed25519 seed via BLAKE2b-512("ZcliWalletSeed" || ed25519_seed). verifies ligerito header proofs and nomt state proofs from zidecar, and cross-checks the chain tip against an independent lightwalletd. trial-decrypts compact blocks locally. the server never learns which addresses belong to you. builds and proves orchard transactions with halo2 locally.

zidecar

the backend. indexes the zcash chain into nomt and generates ligerito proofs over the header chain (gigaproof + tip). serves compact blocks and proofs over gRPC-web. default endpoint: zcash.rotko.net.

zync

shared scanner library. trial decryption, ligerito verification, nullifier tracking, witness construction. used by zcli and zafu.

client-side verification

zcli verifies chain data in three layers before trusting it. each layer builds on the previous one:

layer 1: header chain (ligerito)

before scanning, zcli fetches a combined proof and verifies it locally. the gigaproof (2^26, ~300KB) covers the full historical chain. the tip proof (2^20, ~150KB) covers the gap to current tip, regenerated every block. 

1. splits combined blob into gigaproof + tip proof
2. verifies both ligerito proofs cryptographically
3. checks continuity: tip.start_prev_hash == giga.tip_hash
4. rejects if any check fails

layer 2: state roots (ligerito)

zidecar indexes the zcash chain into nomt merkle trees. a ligerito proof covers the state transition from a known checkpoint to the current height, binding tree_root and nullifier_root to the proven header chain.

layer 3: individual entries (NOMT)

SHA256 merkle proofs verify individual commitments and nullifiers against the proven state roots: 

commitment proofs: cmx exists in tree at position (against tree_root)
nullifier proofs:  nullifier spent/unspent status (against nullifier_root)

if a nullifier is proven spent but the wallet thinks it's unspent, the wallet updates its state. all layers soft-fail gracefully when the server doesn't support the RPCs yet.

what still requires trust

zcash is a proof-of-work chain with no finality. there is no consensus checkpoint a light client can verify independently - the network head is always probabilistic. zcli mitigates this by cross-verifying the tip hash and activation block between zidecar and an independent lightwalletd (default: mainnet.lightwalletd.com, override with --verify-endpoint). if the block hashes diverge, sync aborts.

this means you trust that at least one of your two endpoints is honest about which fork is the real chain. a state-level adversary controlling both could serve a consistent but fabricated fork. this is inherent to PoW light clients without SPV header validation - and zcli does not yet validate proof-of-work on headers. the ligerito proofs guarantee internal consistency of whatever chain you're given, but not that it's the chain with the most work.
